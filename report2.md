## Чек-лист тестирования

## 1\. Проверка присутствия кодов, настроек

- [ ] В настройке CASHBACK\_TYPES присутствуют коды: TRANZ\_1, TRANZ\_2, TRANZ\_8, TRANZ\_17, TRANZ\_19  
- [ ] 1.2. В настройке CASHBACK\_EXCEPTION присутствуют MCC: 4812, 4813, 4814, 4816, 4829, 4900, 6012, 6050  
- [ ] 1.3. В настройке CASHBACK\_VALUE присутствуют MCC: 5339, 5621  
- [ ] 1.4. Возможность добавления/удаления значений в настройках   
- [ ] 1.5. Корректность обработки при пустой/некорректно заполненной настройке (отсутствие значений, дубли)

**A. Пустая настройка (значения отсутствуют)**

**CASHBACK\_TYPES пуст:**

- [ ] Ни по одной транзакции кэшбэк не начисляется (система не падает, не начисляет "по умолчанию")  
- [ ] В логах/мониторинге фиксируется предупреждение о пустой критичной настройке (если предусмотрено)

**CASHBACK\_EXCEPTION пуст:**

- [ ] Транзакции по «чувствительным» MCC (снятие наличных, азартные игры и т.п.) не приводят к аномальным начислениям

Здесь затрудняюсь с чек листом, т.к. не поняла момент с данным полем настройки (CASHBACK\_EXCEPTION). Вопросы: зачем клиенту настраивать это поле? Это нужно только для просмотра кодов МСС \- исключений?

**CASHBACK\_VALUE пуст:**

- По всем транзакциям (в т.ч. по MCC 5339, 5621\) применяется базовая ставка 0,5%, повышенная ставка нигде не срабатывает  
- При пустой настройке проверка MCC возвращает “не найдено”, а не приводит к сбою обработки 

#### **B. Дублирующиеся значения**

- [ ] Дубль кода в CASHBACK\_TYPES (например, TRANZ\_1 указан дважды) \- тип транзакции учитывается один раз, кэшбэк не задваивается  
- [ ] Дубль MCC в CASHBACK\_VALUE \- ставка 5% применяется один раз, а не суммируется/умножается  
- [ ] Один и тот же MCC продублирован с разным регистром/форматом (если формат не строго числовой, напр. с пробелами " 5339") \- проверить нормализацию значения при сравнении

      #### **C. Некорректный формат значений**

- [ ] Нечисловое значение MCC в настройке (например, "abcd" вместо 4812\) \- система не падает, некорректная запись игнорируется или логируется как ошибка конфигурации  
- [ ] MCC с недопустимым количеством разрядов (например, 3 или 5 цифр вместо 4\) \- обрабатывается без падения, транзакция не считается ошибочно попавшей в исключение/повышенную категорию  
- [ ] Пустая строка / null / пробел как отдельный элемент списка \- не приводит к ошибке при обработке транзакций  
- [ ] Лишние пробелы в начале/конце значения (" 4812", "4812 ") \- MCC корректно распознаётся как исключение (тримминг) либо явно не совпадает \- нужно зафиксировать ожидаемое поведение  
- [ ] Разделители между значениями настройки некорректны (двойная запятая, точка с запятой вместо запятой и т.п.) \- не приводит к «слипанию» двух кодов в один

      #### **D. Конфликтующие значения между настройками**

- [ ] Один и тот же MCC присутствует одновременно в CASHBACK\_EXCEPTION и CASHBACK\_VALUE \- зафиксировать фактическое поведение системы (что перевешивает) и сверить с ожиданием бизнеса   
- [ ] Код транзакции присутствует в CASHBACK\_TYPES, но отсутствует в справочнике типов транзакций вообще (несуществующий тип) \- не приводит к ошибке обработки

      #### **E. Изменение настроек «на лету»**

- [ ] Добавление нового MCC в CASHBACK\_EXCEPTION применяется только к новым транзакциям, не затрагивает задним числом уже обработанные записи в справочнике “Кэшбэк по картам”  
- [ ] Удаление MCC из CASHBACK\_VALUE \- по новым транзакциям применяется базовая ставка, ранее начисленный повышенный кэшбэк не пересчитывается  
- [ ] Изменение настройки в момент активной обработки пачки транзакций (гонка) \- используется консистентная версия настройки (нет ситуации, когда часть транзакций в одной пачке обработана по старым значениям, часть \- по новым, без объяснимой причины)

      #### **F. Права доступа и валидация при вводе**

- [ ] Попытка сохранить в настройке значение, не проходящее валидацию формата (если валидация предусмотрена на UI/API уровне) \- система блокирует сохранение с понятной ошибкой, а не сохраняет невалидные значения  
- [ ] Если валидации на вводе нет \- это отдельный риск, который стоит вынести как баг/предложение по доработке

**Замечание по тест-дизайну:** для пунктов A-D желательно завести отдельные тестовые окружения/данные с “испорченными” настройками, а не проверять на проде \- часть этих сценариев (пустая CASHBACK\_EXCEPTION) реально меняет финансовый результат по всем клиентам, если случайно попадёт в боевую конфигурацию.

Также стоит уточнить у аналитика/PM: **предусмотрена ли вообще валидация настроек на уровне ввода**, или это полностью зона ответственности того, кто их заполняет (тогда часть пунктов B–D \- это не баги, а зона риска для документирования, а не для строгого “ожидаемый/фактический результат”).

## 2\. Условия начисления кэшбэка

### **2.1. Признак акции CASHBACK у клиента**

- [ ] Кэшбэк начисляется, если у клиента подключена акция CASHBACK  
- [ ] Кэшбэк НЕ начисляется, если акция CASHBACK не подключена  
- [ ] Проверить поведение системы, если акция CASHBACK отключена клиентом после совершения транзакции, но до её обработки \- начисляется ли кэшбэк исходя из состояния акции на момент покупки, или на момент обработки 

### **2.2. Типы транзакций (CASHBACK\_TYPES)**

- [ ] Кэшбэк начисляется по каждому из типов: TRANZ\_1, TRANZ\_2, TRANZ\_8, TRANZ\_17, TRANZ\_19 (проверить отдельно по каждому)  
- [ ] Кэшбэк НЕ начисляется по типу транзакции, отсутствующему в CASHBACK\_TYPES (например, TRANZ\_3, TRANZ\_18)

### **2.3. Исключения по MCC (CASHBACK\_EXCEPTION)**

- [ ] Кэшбэк НЕ начисляется по каждому MCC из списка исключений (4812, 4813, 4814, 4816, 4829, 4900, 6012, 6050\) \- проверить отдельно по каждому коду  
- [ ] Кэшбэк начисляется по MCC, не входящему в список исключений (например, можем использовать разбиение на классы по категориям и взять по одному коду из каждой категории: Авиалинии \- взять код 3000, Отели \- код 3501, Транспорт и Аренда \- код 4111 и т.д.)

### **2.4. Повышенный кэшбэк по MCC (CASHBACK\_VALUE)**

- [ ] Транзакция по MCC 5339 \- применяется ставка 5%  
- [ ] Транзакция по MCC 5621 \- применяется ставка 5%  
- [ ] Транзакция по MCC, отсутствующему в CASHBACK\_VALUE и CASHBACK\_EXCEPTION \- применяется базовая ставка 0,5%  
- [ ] Проверить приоритет, если MCC одновременно присутствует и в CASHBACK\_EXCEPTION, и в CASHBACK\_VALUE (уточнить у аналитика/PM \- в ТЗ не описано, здесь может быть неоднозначность требований)

## 3\. Проверка расчета суммы кэшбэка

### **3.1. Базовая ставка 0,5%**

- [ ] Покупка 390 руб. → округление до 300 → кэшбэк 1,5 руб.  
- [ ] Покупка 420 руб. → округление до 400 → кэшбэк 2 руб.  
- [ ] Покупка 1100 руб. → округление до 1100 → кэшбэк 5,5 руб.  
- [ ] Покупка ровно 100 руб. → кэшбэк 0,5 руб. (граница “от 100 рублей”)  
- [ ] Покупка 99,99 руб. (меньше 100\) → кэшбэк не начисляется  
- [ ] Покупка 199 руб. → округление до 100 → кэшбэк 0,5 руб.  
- [ ] Покупка 99 руб. →  кэшбэк не начисляется   
- [ ] Покупка 0 руб. / отрицательная сумма (возврат) → кэшбэк не начисляется / корректно обрабатывается

### **3.2. Повышенная ставка 5%**

- [ ] Покупка по MCC 5339 на 740 руб. → округление до 700 → кэшбэк 35 руб.  
- [ ] Покупка по MCC 5621 на сумму, кратную 100 → корректный расчет 5%  
- [ ] Покупка по повышенной категории ровно на 100 руб. → кэшбэк 5 руб.  
- [ ] Покупка по повышенной категории на 99 руб. → кэшбэк не начисляется

### **3.3. Округление суммы**

- [ ] Корректное округление вниз до кратной 100 для различных сумм (99 → 0, 100 →100, 101 → 100, 199 → 100, 200 → 200, 201 → 200, 299 → 200, 300 → 300\)  
- [ ] Округление для сумм с копейками ( например, 350,99 руб.)

### **3.4. Лимит 3 000 руб. в месяц на карту**

- [ ] Сумма кэшбэка за месяц по карте существенно меньше 3000 руб. \- начисляется полностью  
- [ ] Транзакция, после которой накопленный кэшбэк превысит 3000 руб. \- начисляется только часть суммы, не превышающая лимит  
- [ ] Накопленный кэшбэк за месяц уже равен 3000 руб. \- по новой транзакции начисляется 0 руб.  
- [ ] Транзакция, после которой кэшбэк ровно достигает 3000 руб. \- начисляется без превышения  
- [ ] Лимит считается по карте, а не по клиенту (если у клиента несколько карт \- проверить независимый расчет лимита по каждой). **Уточнить у PM возможность начисления кэшбэка до 3000 руб. на каждую карту одного клиента и какие есть ограничения.**   
- [ ] Лимит сбрасывается в начале нового календарного месяца  
- [ ] Транзакции конца месяца/начала следующего месяца корректно относятся к нужному расчетному периоду (граница месяца, часовой пояс)

### **3.5. Признак ПОД/ФТ**

- [ ] У клиента нет отметки ПОД/ФТ \- кэшбэк рассчитывается в обычном порядке  
- [ ] У клиента установлена отметка ПОД/ФТ на момент транзакции → кэшбэк по транзакции \= 0  
- [ ] Отметка ПОД/ФТ установлена ПОСЛЕ совершения транзакции, но до расчета кэшбэка → проверить, какое состояние признака фиксируется (на момент транзакции, согласно ТЗ)  
- [ ] Отметка ПОД/ФТ снята после транзакции \- ранее обнуленный кэшбэк не пересчитывается

## 4\. Учет кэшбэка (операция “Учет начисленного кэшбэка”)

- [ ] Операция автоматически запускается в последний день месяца  
- [ ] Операция НЕ запускается вручную/повторно в тот же день, сумма не дублируется  
- [ ] В выборку попадают операции с датой проводки текущего календарного месяца  
- [ ] В выборку НЕ попадают операции с признаком “Исключение из кэшбэка” \= Да  
- [ ] В выборку НЕ попадают операции с признаком “CashBack выплачен” \= Да  
- [ ] Корректный расчет общей суммы накопленного за месяц кэшбэка (сумма по всем подходящим записям)  
- [ ] Реквизит “Учтенный кэшбэк” заполняется корректной суммой  
- [ ] Формируется платежный документ Дт 70606 / Кт 47422 на правильную сумму  
- [ ] Назначение платежа в документе соответствует шаблону: “Начисление кэшбэка по операциям с использованием пластиковых карт Банка за \[месяц\] \[год\]” с корректной подстановкой месяца и года  
- [ ] Если за месяц нет ни одной операции для учета \- документ не формируется / формируется на нулевую сумму (уточнить ожидаемое поведение)  
- [ ] Корректная работа операции на границе года (декабрь → январь)

### **4.1. Отказ клиента от кэшбэка (отключение акции)**

- [ ] При отказе клиента без выплаты \- накопленный, невыплаченный и НЕ учтенный кэшбэк сгорает  
- [ ] Записям в справочнике “Кэшбэк по картам” проставляется Признак исключения из кэшбэка \= Да  
- [ ] Накопленный, невыплаченный, но уже учтенный (на счете 47422\) кэшбэк также сгорает  
- [ ] Формируется платежный документ Дт 47422 / Кт 70606 на сумму сгоревшего учтенного кэшбэка  
- [ ] Назначение платежа соответствует шаблону: “Списание невостребованного кэшбэка по операциям с использованием пластиковых карт Банка в связи с отказом клиента за период с \[месяц\] \[год\] по \[месяц\] \[год\]” с корректным периодом  
- [ ] Период в назначении платежа корректно определяется (от первого до последнего месяца накопления)  
- [ ] Повторное подключение акции CASHBACK после отказа \- кэшбэк за прошлый период не восстанавливается  
- [ ] Отказ клиента при отсутствии накопленного кэшбэка \- операция не формирует документ / формирует на 0 (уточнить)  
- [ ] Отказ клиента, если часть кэшбэка уже выплачена \- сгорает только невыплаченный остаток

## **5\. Выплата кэшбэка**

- [ ] Доступен к выплате только кэшбэк, учтенный за предыдущие календарные месяцы (текущий месяц недоступен)  
- [ ] Пример из ТЗ: в декабре клиенту доступен кэшбэк за октябрь и более ранние месяцы (проверить \- а за ноябрь? уточнить у бизнес-аналитика формулировку “предыдущие”, т.к. пример упоминает октябрь, а не ноябрь \- потенциальное расхождение в требованиях)  
- [ ] После выплаты у соответствующих записей в справочнике “Кэшбэк по картам” устанавливается признак “Выплачен” \= Да  
- [ ] Выплачивается вся доступная сумма кэшбэка одной операцией/суммой  
- [ ] Клиент не может выбрать частичную сумму к выплате (проверить отсутствие такой возможности в интерфейсе/API)  
- [ ] Клиент не может выбрать счет зачисления \- используется расчетный счет по умолчанию  
- [ ] Формируется платежный документ Дт 47422 / Кт 40802 на корректную сумму и корректный счет клиента  
- [ ] Повторный запрос на выплату при отсутствии доступного кэшбэка (сумма \= 0\) \- корректная обработка (ошибка/отказ без создания документа)  
- [ ] Выплата при наличии нескольких карт клиента \- суммируется кэшбэк по всем картам корректно  
- [ ] Выплата в текущем месяце не включает кэшбэк текущего (еще не учтенного) месяца

## **6\. Интеграционные и сквозные сценарии (E2E)**

- [ ] Полный цикл: транзакция → начисление → учет (конец месяца) → выплата (следующий месяц) \- суммы совпадают на всех этапах.   
- [ ] Несколько транзакций в течение месяца по одной карте \- корректное суммирование с учетом лимита 3000 руб.  
- [ ] Комбинация обычных и повышенных MCC в течение месяца \- раздельный и корректный расчет по каждой  
- [ ] Клиент с несколькими картами и разными признаками (ПОД/ФТ на одной карте, акция отключена на другой) \- независимая обработка по картам  
- [ ] Проверка идемпотентности: повторный запуск операции «Учет начисленного кэшбэка» не приводит к задвоению сумм

## **7\. Негативные и граничные тесты**

- [ ] Транзакция отклонена/отменена \- кэшбэк не начисляется  
- [ ] Возврат товара после начисления кэшбэка \- проверить, аннулируется ли кэшбэк (не описано в ТЗ \- вопрос аналитику)  
- [ ] Некорректные/отсутствующие данные по MCC в транзакции \- обработка без падения системы  
- [ ] Если по одной карте одновременно (в один момент времени) проходит несколько транзакций — лимит 3000 руб. не превышается   
- [ ] Если при создании платёжного документа (проводки Дт/Кт) произошёл сбой \- данные не должны удалиться или задвоиться
